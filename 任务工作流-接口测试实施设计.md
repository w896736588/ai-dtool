# 任务工作流-接口测试实施设计

> 本文档用于指导“任务工作流程页 + 接口测试与覆盖检查”功能的落地实现，偏开发设计与实施拆解。

---

## 一、实施目标

围绕任务清单中的单个任务，新增独立工作流程页，承载以下能力：

1. 展示并复用 TAPD 抓取后的需求 MD
2. 自动创建并维护开发执行 MD
3. 生成需求覆盖检查结果
4. 生成可执行接口测试计划
5. 执行接口测试并保留每次历史记录
6. 提供疑问项、阻塞项、失败摘要，供人工或外部 AI 修复后再次回归

## 五、后端接口设计

### 5.1 工作流基础接口

#### `/api/task/workflow/create_or_get`

入参：

```json
{
  "home_task_id": 123
}
```

返回：

```json
{
  "id": 10,
  "home_task_id": 123,
  "status": "dev_plan_ready",
  "current_stage": "idle",
  "requirement_fragment_id": "req_xxx",
  "dev_plan_fragment_id": "dev_xxx",
  "latest_plan_run_id": 21,
  "latest_test_run_id": 35
}
```

#### `/api/task/workflow/info`

用途：

- 查询工作流详情
- 返回最新覆盖分析摘要、最新测试摘要

### 5.2 开发执行 MD 接口

#### `/api/task/workflow/dev-plan/init`

任务首次进入工作流时自动创建开发执行片段。

#### `/api/task/workflow/dev-plan/info`

返回开发执行片段内容与元数据。

#### `/api/task/workflow/dev-plan/save`

入参：

```json
{
  "workflow_id": 10,
  "content": "# 开发执行说明\n..."
}
```

### 5.3 覆盖分析与测试计划接口

#### `/api/task/workflow/coverage/generate`

入参：

```json
{
  "workflow_id": 10,
  "base_branch": "main"
}
```

返回：

```json
{
  "async_task_id": 201,
  "status": "pending"
}
```

#### `/api/task/workflow/coverage/info`

返回最近一次覆盖分析结果。

#### `/api/task/workflow/test-plan/generate`

入参：

```json
{
  "workflow_id": 10,
  "base_branch": "main"
}
```

#### `/api/task/workflow/test-plan/info`

返回最近一次 `test_plan.json` 与摘要。

### 5.4 测试执行接口

#### `/api/task/workflow/test-run/start`

入参：

```json
{
  "workflow_id": 10,
  "base_branch": "main",
  "run_type": "plan_and_test",
  "env_id": 3
}
```

返回：

```json
{
  "async_task_id": 203,
  "test_run_id": 35,
  "status": "pending"
}
```

#### `/api/task/workflow/test-run/list`

分页返回历史执行记录。

#### `/api/task/workflow/test-run/info`

返回单次执行详情：

- 覆盖分析结果
- 测试计划
- 测试报告
- 摘要 Markdown

#### `/api/task/workflow/test-run/cases`

返回单次执行下的用例结果明细。

#### `/api/task/workflow/test-run/retry`

根据某次历史记录重新执行，必须创建新的 `test_run`。

### 5.5 数据库只读工具接口

#### `/api/task/workflow/db/schema`

查询表结构、索引、主键信息。

#### `/api/task/workflow/db/sample`

查询少量样本数据，必须限制行数。

#### `/api/task/workflow/db/check`

供执行器做数据库只读校验，不用于页面手工 SQL。

## 六、异步任务设计

### 6.1 新增任务类型

```go
const (
    asyncTaskTypeTaskWorkflowCoverageGenerate = "task_workflow_coverage_generate"
    asyncTaskTypeTaskWorkflowTestPlanGenerate = "task_workflow_test_plan_generate"
    asyncTaskTypeTaskWorkflowTestExecute      = "task_workflow_test_execute"
)
```

### 6.2 执行阶段

建议统一阶段名称：

```text
loading_context
checking_coverage
generating_plan
preparing_data
running_cases
checking_db
writing_report
```

### 6.3 日志写入规范

每个阶段至少写：

- 中文和英文阶段名注释
- 当前动作
- 执行结果
- 补充信息

示例：

```text
加载上下文 | 读取需求文档 | 成功 | fragment_id=req_xxx
覆盖检查 | 匹配接口 | 成功 | 命中 5 个接口
测试计划 | 生成用例 | 成功 | 共 12 条
接口测试 | 执行 case-003 | 失败 | code 断言不匹配
数据库校验 | 校验 orders 记录 | 成功 | 命中 1 条
```

## 七、覆盖分析实现设计

### 7.1 输入

- 需求 MD
- 开发执行 MD
- 当前分支 diff
- 接口定义摘要
- 路由 / 控制器摘要
- 数据库 schema 摘要（可选）

### 7.2 输出

结构固定为 `coverage_report.json`：

- `summary`
- `items`
- `questions`
- `blocked`

### 7.3 判断规则

- 每个 `covered / partial / missing` 必须带证据
- 需求不明确进入 `questions`
- 缺少造数能力进入 `blocked`
- 不允许仅凭模型猜测给出最终结论

---

## 八、测试计划实现设计

### 8.1 输入

- 覆盖分析结果
- 需求 MD
- 开发执行 MD
- 接口定义详情
- 环境信息
- 允许调用的前置业务接口列表
- 数据库 schema 摘要

### 8.2 输出

结构固定为 `test_plan.json`，必须包含：

- `coverage_links`
- `preconditions`
- `api_cases`
- `open_questions`
- `blocked_items`

### 8.3 硬约束

- 每条用例必须绑定需求点
- 前置数据优先走业务接口
- 不允许直接写数据库
- 数据库仅做只读校验
- 无法满足条件必须输出阻塞项

---

## 九、测试执行实现设计

### 9.1 执行步骤

1. 读取最近一次测试计划，或在本次流程中生成计划
2. 按 `preconditions` 调用已有业务接口准备数据
3. 按 `api_cases` 调用 `/api/ApiRun`
4. 对断言进行逐条判定
5. 调用数据库只读校验能力验证落库结果
6. 生成 `test_report.json`
7. 写入 `tbl_task_test_run` 与 `tbl_task_test_case_result`

### 9.2 断言类型建议

- `status_code`
- `json_path`
- `json_type`
- `json_not_null`
- `json_contains`
- `response_time`
- `db_exists`
- `db_count`

### 9.3 错误处理原则

- 单条用例失败不应中断整个批次
- 阻塞项导致的跳过要明确标记为 `skipped`
- 结构化结果写入失败时，整个执行记为 `failed`

## 十二、第一期实施顺序

建议按以下顺序开发：

1. 新增工作流程页面壳子
2. 新增 `tbl_task_workflow`
3. 任务创建时自动初始化开发执行 MD
4. 新增 `tbl_task_test_run`
5. 新增覆盖分析接口与异步任务
6. 新增测试计划生成接口与异步任务
7. 新增测试执行接口与历史记录
8. 补充日志、详情页、重跑能力

这样可以优先形成最小可用闭环，并减少并行开发冲突。

## 十三、第一期验收标准

### 功能验收

- 任务清单能进入工作流程页
- 会自动生成开发执行 MD
- 能生成覆盖分析
- 能生成测试计划
- 能执行接口测试
- 能保留每次历史执行记录
- 能查看单次执行详情

### 质量验收

- 历史执行记录不覆盖
- 覆盖分析结果具备证据链
- 前置造数优先走已有业务接口
- 数据库工具无写权限
- 失败用例可查看请求 / 响应 / 断言 / 数据库校验详情

### 稳定性验收

- AI 输出结构不合法时能被拦截
- 单个用例失败不影响其他用例执行
- 页面刷新后可恢复当前任务状态
- SSE 或轮询可稳定获取执行进度

## 十五、结论

本实施设计的重点不是立刻做自动修复，而是把“工作流程页 + 覆盖检查 + 接口测试 + 历史记录”做成稳定底座。

当这套底座跑顺后，无论后续接入自动修复 Agent、扩展 UI 测试，还是沉淀测试平台能力，都会顺很多。
