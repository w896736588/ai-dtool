## 处理流程

### 步骤 1：分析
分析收到的对话历史和工具调用记录：
1. **用户原始问题是什么？** — 理解任务目标和上下文
2. **完整工具调用过程是怎样的？** — 梳理每一步 http_call 的接口路径、传参格式、响应处理逻辑
3. **本次操作模式是否具有通用复用价值？** — 如：查询配置、跨接口数据组装、多步骤流程等
4. **是否已有类似步骤文件？** — 检查 `skills/dtool-butler/index/step.md` 和 `skills/dtool-butler/step/` 目录

输出格式（严格按此格式）：
```
评估结论：[YES/NO]
理由：[简要说明]
步骤文件名建议：[如果YES，建议文件名，如 query_repo_branch.md]
步骤文件描述：[如果YES，一行任务说明]
```

### 步骤 2：生成步骤文件（仅评估为 YES 时执行）

在 `skills/dtool-butler/step/` 下创建 .md 文件，输出格式（严格按此分隔符）：

```
===STEP===
[完整 .md 步骤文件内容]
===INDEX===
- skills/dtool-butler/step/{step_name}.md — {一行任务说明}
```

**步骤文件内容规范**：

```markdown
# {任务标题}

## 任务描述
{一句话说明这个步骤文件能完成什么任务}

## 前置条件
- {需要确认的信息，如：知道仓库名、知道任务状态等}

## 执行步骤

### 步骤1: {步骤名称}
http_call POST {接口路径}
请求体：
{
  "{参数名}": "{参数说明, 变量用{变量}标注}",
  ...
}

#### 响应处理
- 关注字段: `data.{字段名}` → {含义}
- {成功/失败判断条件}

### 步骤2: {步骤名称}
...

## 结果汇总
{最终输出格式说明}
```

**步骤文件要求**：
- 使用 Markdown 格式，结构清晰，参数格式完整
- 不包含会话特定硬编码数据（变量用 `{变量}` 标注）
- 每个 http_call 必须写明：完整的接口路径、请求体 JSON 格式、每个参数的含义
- 多步骤之间有数据依赖时，必须写明依赖关系（如"从步骤1 的 `data.id` 获取"）
- 分页/循环逻辑用自然语言描述（如"每页100条，最多翻100页，空结果则停止"）

### 步骤 3：可用性校验（强制执行，不可跳过）

**生成步骤文件后必须执行以下校验：**

#### 3.1 API 存在性校验（必须执行）
逐项检查文件中引用的所有 API 端点：
1. 用 `file_read` 读取 `skills/dtool-butler/index/apis.md`，确认每个 API 路径在索引中存在
2. 对 apis.md 中未列出的 API，必须用 `file_read` 读取对应的 controller 源码确认路由确实已注册
3. **如果发现引用了不存在的 API → 评估结论必须回退为 NO**

#### 3.2 参数校验（必须执行）
对文件中调用的每个 API：
1. `file_read` 读取对应 controller 源码确认参数名和类型
2. 确认文件中描述的传参与 controller 期望的参数完全一致

#### 3.3 去重检查（必须执行）
1. 检查 `step.md` 索引中是否已有同名步骤文件 → 有则 NO
2. 检查是否已有功能相同的步骤文件 → 有则 NO
3. **严禁创建重复条目**

> ⚠️ **以上校验全部通过后，才能输出 ===STEP=== 和 ===INDEX===。**
> ⚠️ **任何一项校验失败 → 评估结论必须回退为 NO。**

输出校验结果：
```
校验结果：[PASS/FAIL]
- API 存在性：[PASS/FAIL] - [说明]
- 参数匹配：[PASS/FAIL] - [说明]
- 去重检查：[PASS/FAIL] - [说明]
```

### 评估标准
- 当前操作具有通用复用价值，不是一次性查询
- 步骤清晰、参数明确，他人可按照步骤文件复现
- 与已有步骤文件不重复
- **必须通过步骤 3 的全部校验才能判定为 YES**
